# 文档输出模板

生成文档时，按以下模板填充内容。根据实际项目情况增减章节，不要生搬硬套。

---

## 1. intro.md — 项目背景

```markdown
# 项目背景：{项目名}

## 是什么
一句话定义项目。

## 为什么存在
- 解决了什么问题？
- 在它出现之前，人们是怎么做的？有什么不足？

## 目标与设计哲学
- 项目的核心目标
- 设计原则（例如：简洁优先、可扩展、零依赖……）

## 适用场景
- 典型用户是谁？
- 什么情况下应该用它？什么时候不适合用？

## 技术栈
- 语言：
- 框架：
- 关键依赖：

## 项目现状
- 版本/成熟度
- 维护状态、社区规模
- 与同类项目的对比（如有）
```

---

## 2. arch.md — 代码架构

```markdown
# 代码架构：{项目名}

## 整体架构概述
用 2-3 句话描述架构风格（单体/微服务/插件化/管道式/事件驱动……）

## 架构图
（用 Mermaid 或文字流程图表示主要模块关系）

## 模块划分

| 模块名 | 职责 | 对外接口 | 依赖模块 |
|--------|------|----------|----------|
| ...    | ...  | ...      | ...      |

## 核心抽象与设计模式
- 核心类/接口：名称、职责、继承/组合关系
- 使用的设计模式（工厂、策略、观察者、责任链……）

## 外部集成

| 外部服务 | 用途 | 对接模块 | 协议/方式 |
|----------|------|----------|-----------|
| ...      | ...  | ...      | ...       |

## 扩展点
- 项目提供了哪些插件/钩子/回调机制？
- 如何添加新功能而不改动核心？

## 模块依赖方向
描述模块之间的依赖方向，是否存在循环依赖。
```

## 3. dataflow.md — 关键数据流

```markdown
# 关键数据流：{项目名}

## 数据流总览
列出项目的 3-5 条关键数据流及其简要说明。

---

## 数据流 1：{功能名称}

### 触发方式
说明谁/什么触发了这条数据流。

### 流转路径

\```text
[触发] → [模块A.函数1()] → [模块B.函数2()] → [模块C.函数3()] → [输出/副作用]
\```

### 各步骤详解

| 步骤 | 模块/函数 | 输入 | 处理逻辑 | 输出 |
|------|-----------|------|----------|------|
| 1    | ...       | ...  | ...      | ...  |
| 2    | ...       | ...  | ...      | ...  |

### 数据格式变化
说明数据在流转过程中的结构/类型变化。

### 错误处理
异常在哪里被捕获？如何处理？是否有重试/降级/熔断？

---

（对每条关键数据流重复上述结构）

## 跨数据流的共享组件
哪些模块/服务被多条数据流共用？
```

---

## 4. note.md — 代码结构

```markdown
# 代码结构：{项目名}

## 顶层目录一览

| 路径 | 类型 | 用途说明 |
|------|------|----------|
| `src/` | 源码 | ... |
| `tests/` | 测试 | ... |
| ...  | ...  | ... |

## 核心源码目录详解

### `src/{模块名}/`
> 职责：...

| 文件 | 用途 |
|------|------|
| `__init__.py` | 模块入口，导出 ... |
| `core.py` | 核心逻辑，实现 ... |
| ...  | ...  |

（每个重要子目录重复上述结构）

## 关键文件说明

对最重要的 5-10 个文件单独说明：

### `src/engine.py`（举例）
- **作用**：...
- **主要类/函数**：`Engine`, `run()`, `step()`
- **与其他文件的关系**：被 `main.py` 调用，依赖 `config.py`

## 入口点
- 程序主入口：`{文件路径}`
- 库的公开 API 入口：`{文件路径}`
- CLI 入口：`{文件路径}`（如有）
- Web 入口：`{文件路径}`（如有）
```

## 6. sci.md — 科学原理（条件生成）

仅当项目涉及算法/数学/科学内容时生成。不涉及则跳过，并在 `note.md` 末尾注明。

```markdown
# 科学原理：{项目名}

## 涉及的核心领域
列举项目所基于的学科/技术领域。

## 原理详解

### {核心算法/方法名称}

**问题定义**
用数学语言或直觉语言描述这个方法试图解决什么问题。

**核心思想**
该方法的关键洞察是什么？

**数学表述**（如适用）
给出关键公式，并解释每个符号的含义。

**在代码中的体现**
指向具体实现文件和函数，说明理论如何映射到代码。

**参考文献**（如有）
- 论文标题 / arXiv 链接

---

（重复上述结构，覆盖所有核心科学内容）

## 与同类方法的对比

## 局限性与假设
该方法成立的前提假设是什么？在哪些场景下会失效？
```

## 7. dev-guide.md — 开发指南

```markdown
# 开发指南：{项目名}

## 环境搭建
- 系统要求
- 依赖安装步骤
- 配置说明（环境变量、配置文件）

## 本地运行
- 启动命令
- 访问方式
- 常用开发命令

## 测试
- 测试框架
- 运行测试的命令
- 测试覆盖率查看方式

## 调试技巧
- 日志位置与级别配置
- 常用调试手段
- 已知易踩的坑

## 代码规范
- 命名约定
- 目录组织约定
- 提交信息格式（如有）

## 贡献流程
- 分支策略
- PR / Code Review 流程
- 发版流程（如有）
```
